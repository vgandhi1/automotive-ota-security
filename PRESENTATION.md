# Secure Device Provisioning Pipeline — Professional Presentation

---

## Title Slide

**Secure Device Provisioning & Key Injection Pipeline**

Authenticate factory devices with mutual TLS and inject unique cryptographic identity—without exposing keys on the network.

**Target:** Security Manufacturing / Vehicle Production  
**Focus:** PKI, mTLS, factory automation, auditability

---

## Problem Statement

- **Assembly-line devices** need a **unique, strong identity** (private key + certificate) for vehicle-to-cloud and zero-trust operations.
- **Risks** if provisioning is not secured:
  - Keys sent in **plaintext** over the factory network lead to interception.
  - **Rogue or unauthorized devices** could request and obtain identities.
  - **No durable record** of who was provisioned and when leads to a weak audit trail.

**Goal:** Only legitimate factory devices get a permanent identity, keys never travel in the clear, and every provisioning event is logged.

---

## Solution at a Glance

| Aspect | Approach |
|--------|----------|
| **Trust** | Localized **PKI**: Root CA to Factory CA to device/bootstrap certs. |
| **Authentication** | **Mutual TLS (mTLS)**: server requires a valid client (bootstrap) certificate before any key transfer. |
| **Key delivery** | Server generates device key and cert, sends **only over the established TLS 1.3 tunnel**; then **destroys** its copy of the private key. |
| **Storage on device** | Client stores key and cert in a **restricted enclave** (e.g. file with `chmod 400`). |
| **Audit** | **SQLite** audit log: every success (VIN, timestamp, cert serial) and failure (timestamp, reason). |

---

## High-Level Architecture

- **Factory PKI:** Root CA (self-signed) signs Factory CA (intermediate). Factory CA signs device certs, bootstrap certs, and server cert.
- **Device** holds a **bootstrap certificate** (pre-installed in manufacturing) and connects over **mTLS** to the Factory Server.
- **Factory Server** requires client certificate, generates per-device ECC key and cert, signs with Factory CA, returns key and cert over TLS, logs success/failure, and does not retain the device private key.
- **Device** stores the returned identity in an **enclave** (e.g. `chmod 400` files).

---

## Security: Threat Model and Mitigations

| Threat | Mitigation |
|--------|------------|
| **Network sniffing** | All key material is sent only inside a mutually authenticated TLS 1.3 tunnel. |
| **Rogue device** | Server requires a client certificate; connections without a valid bootstrap cert (chaining to Root CA) are rejected. |
| **Insider / server compromise** | Server deletes the device private key from memory after sending it; audit log records every successful issuance (VIN, time, cert serial) and failures. |

---

## Implementation Highlights

- **Phase 1 — CA:** Root CA + Factory CA + secure vault layout; bootstrap and server certs generated by `ca/scripts/setup_ca.py`.
- **Phase 2 — mTLS:** FastAPI server with TLS configured to require client certificates; device client uses bootstrap cert and Root CA for verification.
- **Phase 3 — Key injection:** Server generates secp256r1 key and CSR per VIN, signs with Factory CA (SHA-256 ECDSA), returns key and cert over mTLS; client writes to enclave with chmod 400.
- **Phase 4 — Audit and scale:** SQLite audit DB (success + failure); Dockerfile for containerized deployment.

---

## Business Value

- **Aligns with production security:** Integrates security controls into vehicle/device production (e.g. assembly line).
- **Foundation for IoT/vehicle-to-cloud:** Strong, unique identity per device enables secure cloud connectivity and zero-trust models.
- **Auditability:** Immutable log of what was provisioned and when—support for compliance and incident response.
- **Repeatable and automatable:** Scripts and containerization support consistent, scalable factory rollout.

---

## Demo / How to Run

1. **Setup (once):**  
   `python3 ca/scripts/setup_ca.py`

2. **Start server (mTLS on 8443):**  
   `python3 -m uvicorn server.factory_server:app --host 0.0.0.0 --port 8443 --ssl-keyfile certs/server/server.key --ssl-certfile certs/server/server.crt --ssl-cert-reqs 2 --ssl-ca-certs ca/root_ca/ca.crt`

3. **Provision a device:**  
   `python3 -m client.device_client VIN12345 --url https://localhost:8443`

4. **Result:** Device identity appears under `certs/device_enclave/` (e.g. `VIN12345.key`, `VIN12345.crt`) with restricted permissions.

## Summary

- **Localized PKI** and **mTLS** ensure only authorized factory devices receive a permanent identity.
- **Keys are never transmitted in plaintext**; server does not retain device private keys after transfer.
- **Audit logging** and **containerization** support compliance and deployment at scale.
